# WebRTC API

The WebRTC channel provides lower-latency real-time communication for live conversation sessions. Binary audio travels over an unordered DataChannel, while all control messages use the same JSON protocol as the [WebSocket API](./websocket).

For a step-by-step usage guide, see [WebRTC Channel](../guide/webrtc).

## Signaling Endpoint

### POST /api/webrtc/offer

Exchange an SDP offer for an SDP answer. The server gathers all ICE candidates before responding (no trickle ICE).

**No authentication required** — the session is authenticated with an API key via the `auth` message sent over the `control` DataChannel after the WebRTC connection is established.

#### Client Prerequisite

Before calling this endpoint, the client **must** create two named DataChannels on the `RTCPeerConnection`:

| Channel name | `ordered` | `maxRetransmits` | Purpose |
|---|---|---|---|
| `control` | `true` | — (unlimited) | JSON messages — identical to WebSocket protocol |
| `audio` | `false` | `0` | Binary audio frames — user voice in, AI voice out |

Creating the DataChannels before generating the offer ensures they are included in the SDP and the server opens them via `ondatachannel`.

```javascript
const pc = new RTCPeerConnection();
pc.createDataChannel('control', { ordered: true });
pc.createDataChannel('audio', { ordered: false, maxRetransmits: 0 });

const offer = await pc.createOffer();
await pc.setLocalDescription(offer);
```

#### Request

```http
POST /api/webrtc/offer
Content-Type: application/json
```

```json
{
  "sdpOffer": "<SDP offer string from RTCPeerConnection.localDescription.sdp>"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `sdpOffer` | `string` | Yes | SDP offer generated by the client `RTCPeerConnection` |

#### Response

```http
HTTP/1.1 200 OK
Content-Type: application/json
```

```json
{
  "sdpAnswer": "<SDP answer containing all server ICE candidates>"
}
```

| Field | Type | Description |
|---|---|---|
| `sdpAnswer` | `string` | Complete SDP answer. Pass to `pc.setRemoteDescription({ type: 'answer', sdp: sdpAnswer })` |

#### Error Responses

| Status | Description |
|---|---|
| `400 Bad Request` | `sdpOffer` is missing or not a string |
| `500 Internal Server Error` | Peer connection creation failed or ICE/signaling error |

## Control Channel Protocol

All JSON messages on the `control` DataChannel use the same format and type names as the WebSocket API. See [WebSocket API](./websocket) for the full message reference.

### Session ID

The session ID is returned in the `auth` response on the `control` channel (same as WebSocket):

```json
{
  "type": "auth",
  "requestId": "req-1",
  "success": true,
  "sessionId": "session-uuid",
  "projectSettings": { ... }
}
```

Include `sessionId` in every subsequent message.

### Rate Limiting

Auth messages on the control channel are rate-limited to **10 attempts per 15 minutes per IP** (the client IP captured during the HTTP signaling request). Exceeding the limit sends an error and closes the DataChannels.

## Audio Channel Binary Frame Format

All audio data on the `audio` DataChannel — both inbound (user voice → server) and outbound (server AI voice → client) — uses the same binary frame format:

```
Offset  Size     Description
──────  ───────  ─────────────────────────────────────────────
0       2 bytes  uint16 little-endian: byte length of turnId
2       N bytes  turnId encoded as UTF-8
2+N     rest     raw audio data
```

### Encoding (client → server, user voice)

The `turnId` in user voice frames is the `inputTurnId` returned by the `start_user_voice_input` response.

```javascript
function encodeAudioFrame(turnId, audioBuffer) {
  const turnIdBytes = new TextEncoder().encode(turnId);
  const header = new ArrayBuffer(2);
  new DataView(header).setUint16(0, turnIdBytes.length, true); // LE
  return concatBuffers([header, turnIdBytes.buffer, audioBuffer]);
}
```

### Decoding (server → client, AI voice)

The `turnId` in AI voice frames is the `outputTurnId` for the current generation turn.

```javascript
audioChannel.onmessage = async (event) => {
  const buf = event.data instanceof ArrayBuffer
    ? event.data
    : await event.data.arrayBuffer();
  const view = new DataView(buf);
  const turnIdLength = view.getUint16(0, true);
  const turnId = new TextDecoder().decode(new Uint8Array(buf, 2, turnIdLength));
  const audio  = buf.slice(2 + turnIdLength);
  // use turnId + audio...
};
```

## Voice Input Flow

Voice input combines control-channel signaling with audio-channel binary streaming:

1. Send `start_user_voice_input` over `control` → receive `inputTurnId`
2. Send binary audio frames over `audio` (each frame carries `inputTurnId`)
3. Send `end_user_voice_input` over `control`

See [WebSocket API — Voice Input Flow](./websocket#voice-input-flow) for the signaling message schemas (identical for WebRTC).
